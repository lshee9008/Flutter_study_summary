- 플러터의 상태관리 3대장 : Provider / Riverpod / BLoC

# Provider (ChangeNotifier 기반)

- 장점
    - 러닝커브 가장 낮음
    - Flutter 공식 추천 → 래퍼런스 많음
    - 소규모 앱, 빠른 프로토타이핑에 최고
    - 구조 단순 → 디버깅 쉬움
- 단점
    - ChangeNotifier 특성상
        - 불필요한 rebuild 발생 쉬움
        - 상태가 커질수록 관리 난이도 급상승
    - 비즈니스 로직이 UI랑 섞이기 쉬움
    - 테스트 코드 작성이 귀찮음
- 언제 써야 하나???
    - MVP / 개인 프로젝트
    - 화면 수 적은 앱
    - “일단 빨리 만들어야 함” 이 목표일 떄
    - 상태 복잡도 낮음 (토글, 리스트, 간단한 API)
- 한 줄 요약
    - 작고 빠르게 → Provider

# Riverpod (Provider의 진화형)

- 장점
    - Context 필요 없음
    - 컴파일 타임 안전성 높음
    - 전역 상태 관리 깔끔
    - 상태 분리 강력 (Provider 조합)
    - 테스트 작성 매우 쉬움
    - async 상태 관리 (AsyncValue) 깔끔
- 단점
    - Provider 보다 러닝커브 있음
    - 코드 양이 조금 늘어남
    - 초보자에게 개념이 다소 추상적
- 언제 써야 하나
    - 중~대규모 앱
    - API / 비동기 로직 많음
    - 상태 의존 관계 복잡
    - 장기 운영 앱
    - Provider 쓰다 스트레스 받을 때
- 한 줄 요약
    - Provider의 상위호환, 실무 표준

# BLoC

- 장점
    - 아키텍처 가장 명확
    - UI ↔ 비즈니스 로직 완전 분리
    - 상태 흐름 추적 최고
    - 테스트 친화적
    - 팀 협업에 매우 강함
- 단점
    - 코드량 많음
    - 러닝커브 큼
    - 작은 앱엔 과함
    - 보일러플레이트 느낌 있음
- 언제 써야 하나
    - 대규모 팀 프로젝트
    - 복잡한 상태 흐름
    - 명확한 이벤트 기반 시스템
    - 금융 / 커머스 / 엔터프라이즈 앱
- 한 줄 요약
    - 규칙이 필요하면 BLoC

# 한 눈에 비교표

| 기준 | Provider | Riverpod | BLoC |
| --- | --- | --- | --- |
| 난이도 | ⭐ | ⭐⭐ | ⭐⭐⭐ |
| 코드량 | 적음 | 중간 | 많음 |
| 성능 | 보통 | 좋음 | 좋음 |
| 테스트 | 불편 | 쉬움 | 매우 쉬움 |
| 규모 | 소형 | 중~대형 | 대형 |
| 실무 선호 | 낮음 | ⭐⭐⭐⭐ | ⭐⭐⭐ |

# 추천 가이드

- 혼자 / 사이트 프로젝트 → Riverpod
- 빠른 MVP → Provider
- 팀 프로젝트 / 회사 → BLoC or Riverpod
- FastAPI + Flutter + AI 서비스(내가 자주 사용하는 아키텍처) → Riverpod

# 개인적인 결론

- Provider ❌ (레거시 느낌)
- **Riverpod ✅ 메인**
- BLoC ✅ 팀/엔터프라이즈
